> Audience: site owner. Region: Iloilo / Western Visayas, Philippines. Reference plan: `docs/superpowers/plans/2026-05-16-arsd-co-seo-implementation.md` (Tasks 1–16, all PASS at time of writing). Production URL: `https://www.arsd.co`.

## Phase 1 — Day 0 Setup (no data exists yet)

### 1.1 Domain property vs. URL-prefix property

GSC offers two kinds of "property":

- **Domain property** — covers `arsd.co` and **every** subdomain (`www.arsd.co`, future `blog.arsd.co`, etc.) and **both** protocols (`http://`, `https://`). Verified via DNS.
- **URL-prefix property** — covers only the exact prefix you enter (e.g., `https://www.arsd.co/`). You would need a separate property for `arsd.co` without `www`, for `http://`, and for any subdomain.

**Recommendation: Domain property.** Reasons:
- arsd.co is a single-site business; future subdomain work (e.g., a Help Center) will be covered automatically.
- Consolidates Performance + Indexing data; URL-prefix splits the data across properties and you lose the unified view.
- The only downside is DNS verification is mildly more involved than HTML-tag verification, but you only do it once.

You may **additionally** create a URL-prefix property for `https://www.arsd.co/` later if you ever need to compare `www` vs. non-`www` traffic. For now, Domain only.

### 1.2 DNS TXT record verification walkthrough

Order of operations:

1. Open `https://search.google.com/search-console` and sign in with the Google account you want to own the property long-term. **Do not** use a personal account that may become inactive — use one tied to the business.
2. On the welcome screen, click **Add property**. Choose the left card, **Domain**.
3. Enter `arsd.co` (no `https://`, no `www.`). Click **Continue**.
4. GSC displays a TXT record value that starts with `google-site-verification=` followed by a long string. Copy the entire string including `google-site-verification=`.
5. Log into the DNS provider that hosts `arsd.co`'s DNS (Cloudflare, Vercel, GoDaddy, Namecheap, etc.). Open the DNS records / zone editor for `arsd.co`.
6. Create a new record:
   - **Type:** `TXT`
   - **Name / Host:** `@` (this means the apex domain itself; some providers want it blank, some want `arsd.co`)
   - **Value / Content:** the full `google-site-verification=...` string
   - **TTL:** default (typically 300–3600 seconds) is fine
7. Save the record. **Do not delete any existing TXT records** (SPF, DMARC, etc.) — add a new one alongside them.
8. Wait 5–15 minutes. Then back in GSC click **Verify**. If it fails, wait another 30 minutes and try again — TXT propagation can be slow.
9. Once verified, GSC will say "Ownership verified". **Leave the TXT record in place permanently** — if you delete it, GSC will eventually un-verify the property.

### 1.3 Submit the sitemap

The sitemap was generated by **Task 3** (`src/app/sitemap.ts`). It is served at `https://www.arsd.co/sitemap.xml` and contains **27 URLs** (5 marketing pages + 22 project detail pages).

1. Before submitting, open `https://www.arsd.co/sitemap.xml` in a browser. Confirm it loads as XML, not 404, not HTML error page. You should see `<urlset>` and `<url>` tags with `<loc>` entries.
2. In GSC's left nav, click **Indexing → Sitemaps**.
3. In the "Add a new sitemap" field, type `sitemap.xml` (no leading slash, no domain — GSC prefixes the property root automatically).
4. Click **Submit**.
5. Status should change to "Success" within a few minutes. "Discovered URLs" will say `27` (or close to it).
6. If status is "Couldn't fetch", recheck the sitemap URL is reachable and the `robots.txt` from Task 2 references it (open `https://www.arsd.co/robots.txt` and confirm the line `Sitemap: https://www.arsd.co/sitemap.xml` is present).

### 1.4 Request indexing for priority pages

The pre-implementation audit found that Google had indexed only 2 URLs (`/` and `/sign-up`). After Tasks 4 and 9, `/sign-up` is now `noindex` and the four marketing pages should be discoverable. Don't wait for natural recrawl — request indexing manually for the high-priority pages.

In GSC's left nav, click **URL Inspection** (top of the left rail, with a magnifying-glass icon). For each URL below, paste the full URL into the top search bar and hit Enter, then click **Request indexing** if the page says "URL is not on Google" or "URL is available to Google":

1. `https://www.arsd.co/` (homepage — re-request to refresh after the Task 9 H1/metadata changes)
2. `https://www.arsd.co/about-us`
3. `https://www.arsd.co/our-services`
4. `https://www.arsd.co/projects`
5. `https://www.arsd.co/contact-us`
6. Two or three of the most important `/projects/[id]` detail pages (pick the most flagship/recent projects)

Quotas: Google limits "Request indexing" to ~10 per day per property. Don't burn it on low-priority pages. Each request takes ~1 minute to process — wait for the confirmation toast before moving to the next URL.

### 1.5 Confirm `robots.txt` is not blocking anything critical

1. In GSC left nav, click **Settings** (gear icon at the bottom of the left rail), then under "Crawling" click **Open report** next to **robots.txt**.
2. You should see one entry: `https://www.arsd.co/robots.txt`, status "Fetched", with the file contents displayed.
3. Confirm the **Allow / Disallow** rules match Task 2's intent:
   - `Disallow: /sign-up`, `/sign-in`, `/dashboard`, `/pending-approval`, `/api/`, `/auth/` — these should all be **disallowed**
   - The marketing routes (`/`, `/about-us`, `/our-services`, `/projects`, `/contact-us`) should NOT appear in any Disallow line
   - `Sitemap: https://www.arsd.co/sitemap.xml` should be the last line
4. If anything looks wrong, **do not edit in GSC** (GSC's robots.txt editor is a tester, not the live file). The live file is generated by `src/app/robots.ts` — flag as a TODO for a coding session.

### 1.6 Alert preferences

1. In GSC, click the **gear icon (Settings)** at the bottom of the left rail.
2. Click **Users and permissions**. Confirm your account is listed as **Owner**.
3. Top-right, click your avatar → **Preferences**. Confirm the email shown is the inbox you actually read.
4. Back to the property, click **Settings → Property settings**. Scroll to **Email preferences**. Enable **all** notification types:
   - Coverage issues
   - Manual actions
   - Security issues
   - Page experience / Core Web Vitals issues
   - Schema (rich results) issues
   - Performance / large drops in clicks or impressions

GSC will email you when any of these trigger. The Performance alert is the most important — it fires on statistically significant drops, which is your early warning for a manual action or algorithmic hit.

### 1.7 What to expect: 72 hours, 7 days, 30 days

- **First 72 hours.** Property verified. Sitemap status = Success. Indexing requests processed. URL Inspection for the homepage shows "URL is on Google" (it was already indexed). The four previously-missing pages will likely flip from "URL is not on Google" to "URL is on Google" within 24–72 hours given the explicit indexing requests. **No Performance data yet** — GSC doesn't populate that report until impressions accumulate.
- **First 7 days.** Performance report starts showing impressions, possibly clicks. Most impressions will be for brand queries ("arsd", "arsd construction", "arsd iloilo"). Indexing → Pages will show 5–10 indexed pages. Expect 5–15 sitemap URLs to remain in "Discovered – currently not indexed" — Google often delays indexing pages with thin signals, especially `/projects/[id]` detail pages.
- **First 30 days.** Performance report has enough data for meaningful query analysis. Total impressions for a site this size will likely be in the low thousands for the month (small site, low-volume queries, geo-restricted intent — this is normal for regional construction services in Iloilo). Most clicks will still be brand queries. Non-brand impressions (e.g., "construction company iloilo", "PCAB contractor iloilo") will start appearing but may have low CTR until the pages rank past position 10. Core Web Vitals should have enough field data by day 28–35 to start showing real-user metrics.

**Do not panic if:**
- Total impressions are < 500 in the first 30 days (small regional site, low search demand)
- Some `/projects/[id]` pages are "Discovered – currently not indexed" (Google delays indexing thin pages; this resolves as the pages accrue signals)
- Average position is in the 20s or 30s for most non-brand queries (rankings climb over 60–90 days, not days)

## Phase 2 — Initial Deep-Dive (after ~14 days of data)

### 2.1 Overview

- (a) **What it is.** The landing screen for the property. Three big cards: Performance summary, Indexing summary, Experience summary. Plus a fourth "Enhancements" card if structured data was detected.
- (b) **Where to click.** **Overview** (top of the left rail, house icon).
- (c) **What to look for.** Glance only — the Overview is a triage screen, not an analysis screen. Confirm: Performance card shows non-zero clicks/impressions; Indexing card shows ~25–27 indexed pages (the 27 sitemap URLs minus any holdouts); Experience card shows Core Web Vitals data appearing (or "Not enough data" if traffic is still very low); Enhancements card lists FAQ, LocalBusiness, Service, BreadcrumbList without errors.
- (d) **How to act.** Whichever card shows a red triangle / "Issues" badge — go directly to that report. If everything is clean here, skip to Performance.

### 2.2 Performance → Search results

- (a) **What it is.** The single most valuable report. Shows queries, pages, countries, devices, and search-appearance breakdowns of impressions, clicks, CTR, and average position from Google Search.
- (b) **Where to click.** **Performance → Search results**.
- (c) **What to look for.** Default view is last 3 months. Switch to **Last 28 days** for current-state analysis. Confirm all four toggles at the top are enabled (Total clicks, Total impressions, Average CTR, Average position).
   - **Queries tab.** Top brand queries (`arsd`, `arsd construction`, `arsd iloilo`) should dominate. Non-brand wins to look for: location + service combinations (`construction iloilo`, `general contractor iloilo`, `pcab contractor iloilo`, `architectural design iloilo`, `warehouse construction philippines`). Red flags: queries with position 5–15 and CTR below 1% (means title/meta are not earning the click); queries with 100+ impressions and zero clicks (intent mismatch or unappealing snippet).
   - **Pages tab.** Five marketing pages should have impressions. `/our-services` and `/projects` should be earning impressions on non-brand queries. Red flag: a marketing page with zero impressions after 14+ days — means it's either not indexed or has no targeted query.
   - **Countries tab.** PH should be > 80% of clicks. Significant impressions from non-target countries (US, India) usually mean a non-localized query is matching — not necessarily bad, but not your revenue target.
   - **Devices tab.** PH market is mobile-dominant — expect 60–80% mobile impressions. If desktop dominates, something is wrong (or your only visitors are dashboard users, which is a different audience).
   - **Search appearance tab.** Look for "Rich results" entries: FAQ, Breadcrumbs, Sitelinks. If FAQ doesn't appear after 30 days, the FAQPage JSON-LD (Task 13) may not be parsing — cross-check in **Enhancements → FAQ** (section 2.9).
   - **Dates tab.** Look for sudden cliffs (>30% drop in clicks over 3 days) — that's a manual-action or algorithmic-hit signal.
- (d) **How to act.**
   - Low CTR on a high-impression query → revise the page's title tag and meta description (ties to **Task 9** — per-page metadata). Target the exact query phrasing in the title and a benefit-led sentence in the meta description.
   - High-impression query with no matching page → write a new page or add a section to an existing page targeting the query. This becomes a new content task; record it as a backlog item.
   - A page earning impressions on a query that doesn't match the page's actual service → intent mismatch. Either retarget the page or accept the irrelevant traffic (don't suppress it; just don't optimize for it).
   - Geo distribution skewed away from PH → revisit `LocalBusiness` JSON-LD (**Task 11**) and confirm `addressCountry` is `PH` and `addressRegion` is `Western Visayas` (or `Iloilo`). If those are wrong, name a new task to correct them.

### 2.3 Indexing → Pages

- (a) **What it is.** The authoritative "what's actually in Google's index from your site" report. Splits URLs into "Indexed" (good) and "Not indexed" (with reasons).
- (b) **Where to click.** **Indexing → Pages**.
- (c) **What to look for.**
   - **Indexed count.** Target = 25–27 (the 5 marketing + 22 project detail pages from the Task 3 sitemap). Acceptable = 20+. Below 15 = problem.
   - **Not indexed reasons.** Click each reason to see which URLs fell into that bucket. Reasons you want to see (these are correct):
     - "Excluded by 'noindex' tag" — should list `/sign-up`, `/sign-in`, `/dashboard/*`, `/pending-approval` (this is **Task 4** working as intended)
     - "Blocked by robots.txt" — should list `/api/*`, `/auth/*` (this is **Task 2** working as intended)
   - **Reasons that indicate problems:**
     - "Discovered – currently not indexed" — Google found the URL but hasn't crawled it. Normal for thin `/projects/[id]` pages early on; should resolve in 30–60 days. If it persists past 90 days for the same URL, the page has weak signals (no internal links, no external links, very thin content).
     - "Crawled – currently not indexed" — Google crawled the page but chose not to index. Worse than "Discovered". Means Google looked and was unimpressed. Audit the page: is it thin? Duplicative of another page? Missing structured data?
     - "Page with redirect" — only acceptable for `http://arsd.co/*` → `https://www.arsd.co/*` and non-`www` → `www` redirects. Any marketing-page-to-marketing-page redirect is a bug.
     - "Soft 404" — page returns 200 but Google thinks it's empty. Common on dynamic detail pages with missing data.
     - "Duplicate without user-selected canonical" — two URLs that Google considers the same, and your canonical tag wasn't decisive. The Task 9 canonical tags should prevent this; if it appears, audit the canonical on the affected pages.
- (d) **How to act.**
   - "Discovered – currently not indexed" on `/projects/[id]` pages → add internal links from `/projects` and `/our-services` to specific project pages; consider adding a short paragraph of unique content per project (ties to **Task 12** — projects are DB-driven from Supabase; a new operator task is needed: enrich project records in Supabase with per-project descriptions).
   - "Crawled – currently not indexed" on a marketing page → check that page in URL Inspection, look at the "Last crawl" snapshot, and confirm the rendered HTML contains the JSON-LD (Tasks 10, 11, 13). If JSON-LD is missing from the rendered output, it's a hydration/SSR bug — flag as a coding TODO.
   - "Page with redirect" on an unexpected URL → find where the redirect is set (Next.js `redirects()` in `next.config.js`, or a Vercel rewrite, or a middleware redirect). Flag as a coding TODO.
   - "Soft 404" → audit the page rendering. For `/projects/[id]`, this likely means a missing project record. Cross-check the Supabase `projects` table.
   - Total indexed count below 20 → request indexing for the missing URLs via URL Inspection, then revisit in 14 days.

### 2.4 Indexing → Sitemaps

- (a) **What it is.** Status report for every sitemap you've submitted.
- (b) **Where to click.** **Indexing → Sitemaps**.
- (c) **What to look for.** One row: `sitemap.xml`, status `Success`, "Discovered URLs" = 27 (or close — Google's count can lag the actual file by a few). "Last read" should be within the last 7 days. Red flags: status `Has errors`, `Couldn't fetch`, or "Discovered URLs" stuck at a number much lower than 27.
- (d) **How to act.**
   - `Couldn't fetch` → open `https://www.arsd.co/sitemap.xml` in a browser. If it loads, click the sitemap row in GSC and hit the refresh icon. If it still fails, the sitemap-route caching may need a forced rebuild — flag as a coding TODO (re-deploy or check `src/app/sitemap.ts` revalidate behavior).
   - `Has errors` → click the row to see specific errors. Most commonly malformed `<lastmod>` dates or invalid URL characters. Flag the specific URLs as a TODO for `src/app/sitemap.ts`.
   - "Discovered URLs" much less than 27 → the sitemap is parsing but Google is rejecting some URLs (typically because they're already disallowed in robots.txt, redirected, or return non-200). Click into the specifics; this is normal noise up to ~10% of URLs.

### 2.5 Indexing → Video pages

- (a) **What it is.** Indexing report for video-bearing pages (requires `VideoObject` schema or detected video embeds).
- (b) **Where to click.** **Indexing → Video pages**.
- (c) **What to look for.** N/A. arsd.co has no video content. The report should be empty or show "Not enough data".
- (d) **How to act.** Skip. Revisit only if a future content initiative adds video (e.g., project-walkthrough videos), at which point add `VideoObject` JSON-LD as a new task.

### 2.6 Experience → Core Web Vitals

- (a) **What it is.** Real-user (Chrome User Experience Report / CrUX) field data for three metrics: LCP (Largest Contentful Paint), INP (Interaction to Next Paint), CLS (Cumulative Layout Shift). Split into Mobile and Desktop.
- (b) **Where to click.** **Experience → Core Web Vitals**.
- (c) **What to look for.** Each device shows "Good", "Needs improvement", or "Poor" URL counts. The pre-implementation audit couldn't compute these because there was no traffic — expect "Not enough data" until ~30 days post-launch for a small regional site. Once data appears: target = all URLs in "Good". Red flag = any URLs in "Poor", especially Mobile LCP > 2.5s (PH market is mobile-heavy and bandwidth-variable).
- (d) **How to act.**
   - Poor Mobile LCP → the hero image or hero video is the usual culprit. Audit `src/components/hero.tsx`; confirm the hero image uses `next/image` with `priority`, has correct `sizes`, and is served as WebP/AVIF. Flag as a coding TODO if not.
   - Poor INP → JavaScript main-thread blocking on interaction. Most likely on dashboard pages (which are noindex anyway — but still worth fixing for users). For marketing pages, audit any heavy client components.
   - Poor CLS → image dimensions not reserved, late-loading font, or animation that shifts layout. Check the hero, image galleries on `/projects/[id]`, and FAQ accordion (**Task 13** — confirm Radix Accordion doesn't introduce layout shift on open).

### 2.7 Experience → HTTPS

- (a) **What it is.** Confirms HTTPS coverage across indexed URLs.
- (b) **Where to click.** **Experience → HTTPS**.
- (c) **What to look for.** "HTTPS URLs" count should equal Indexing → Pages "Indexed" count. Any "HTTP URLs" is a red flag — means an `http://` version slipped into the index.
- (d) **How to act.** Any HTTP URLs → confirm the `http://arsd.co/*` → `https://www.arsd.co/*` redirect is in place at the DNS/hosting layer (Vercel handles this automatically; verify it wasn't disabled). Flag as a coding TODO if missing.

### 2.8 Experience → Mobile Usability

- (a) **What it is.** Legacy report covering mobile rendering issues (tap targets, viewport, content sizing). Officially deprecated in late 2023, but some properties still see the link.
- (b) **Where to click.** **Experience → Mobile Usability** (if visible; otherwise skip).
- (c) **What to look for.** If still shown: "Usable" page count should equal indexed count. Any "Not usable" entries with specific issues (small tap targets, text too small, content wider than screen).
- (d) **How to act.** Issues here usually mean the page has horizontal overflow on mobile, a button under 48×48px, or fixed pixel font size. Manually open the page on a phone (real device, not desktop devtools emulator) and confirm. Flag specific files as a coding TODO. Even though the report is deprecated, the underlying UX problem is still real.

### 2.9 Enhancements (structured-data reports)

The implementation added five JSON-LD types. Each gets its own report in GSC under **Enhancements** (the section will be empty until Google parses the markup, typically 1–4 weeks after first crawl). For each report, the structure is the same:

- (a) **What it is.** Per-schema-type validation report.
- (b) **Where to click.** **Enhancements → [schema type]**.
- (c) **What to look for.** Two columns: "Valid" (good) and "Invalid" (errors) or "Valid with warnings". You want all items in Valid with zero warnings.
- (d) **How to act.** Click any error/warning row to see the affected URLs and the specific issue (missing required field, wrong type, malformed value). Then tie to the source task:

  - **FAQPage** → markup added in **Task 13** on `/` and `/our-services`. Errors usually mean question/answer text doesn't match between the JSON-LD and the rendered FAQ (Google requires text-match integrity). If a warning appears, re-verify the text in the JSON-LD blob matches the visible FAQ verbatim (the task notes "text-match integrity preserved (programmatic check passed)" — a regression would mean someone edited the visible copy without updating the JSON-LD).
  - **LocalBusiness** → markup added in **Task 11** in the root layout. Common warnings: missing `geo` (latitude/longitude), missing `openingHours`, missing `priceRange`. The plan memo lists "office geo coords for LocalBusiness schema" as a pending operator action — close that gap if the warning appears.
  - **Organization** → markup added in **Task 10**. Usually clean. If warnings appear, the typical cause is missing or malformed `sameAs` social profile URLs.
  - **Service** (×5) → markup added in **Task 11** on `/our-services`. Warnings usually mean missing `areaServed` or `provider` references. Cross-check against `src/data/services-data.ts`.
  - **BreadcrumbList** → markup added in **Task 12** for `/projects/[id]` pages. Errors usually mean a `position` field is missing or the breadcrumb name doesn't match the page H1.

If a schema type that should exist doesn't appear in Enhancements after 30 days → the markup isn't being parsed. Open the page, "View Source", and confirm a `<script type="application/ld+json">` block is present in the rendered HTML. If it's only in the client bundle and not the SSR HTML, that's a hydration bug — flag as a coding TODO.

### 2.10 Links

- (a) **What it is.** Inventory of inbound (external) and internal links Google has discovered.
- (b) **Where to click.** **Links** (chain icon in left rail).
- (c) **What to look for.**
   - **External links → Top linked pages.** Homepage will lead. Look for any internal pages with surprisingly many external links — could be link-bait coverage worth amplifying.
   - **External links → Top linking sites.** Expect a few directory listings (Dun & Bradstreet has already been spotted; PhilGEPS likely too; local business directories). Red flag: spammy domains or PBN-style sites linking with money-keyword anchor text — that's a negative-SEO indicator, not a win.
   - **External links → Top linking text.** Mostly brand variants ("ARSD", "ARSD Construction"). Red flag: large counts of commercial anchors ("construction company iloilo", "general contractor philippines") from low-quality sources.
   - **Internal links → Top linked pages.** Homepage and `/our-services` should dominate. Any `/projects/[id]` page with 0 internal links is a coverage gap.
- (d) **How to act.**
   - Surprising external referrers → consider a follow-up "thank you" email or a backlink-relationship note for future outreach.
   - Spammy referrers → Google generally ignores low-quality links. Only disavow if a Manual Action specifically cites unnatural links (see 2.11).
   - `/projects/[id]` pages with zero internal links → add internal links from `/our-services` (service-to-relevant-project), homepage featured-projects section, and a related-projects block at the bottom of each project page. Name this as a new task: "Add related-projects internal links to project detail pages."

### 2.11 Security & Manual actions

- (a) **What it is.** Two reports: **Manual actions** (human-issued penalties by Google's webspam team) and **Security issues** (malware, hacked content, social engineering warnings).
- (b) **Where to click.** **Security & Manual actions → Manual actions**, then **→ Security issues**.
- (c) **What to look for.** Both should read **"No issues detected"**. Anything else is an emergency.
- (d) **How to act.**
   - Manual action issued → read the action description carefully, fix the underlying issue (most common for a small site would be "Thin content" or "User-generated spam"), then submit a Reconsideration Request via the same screen. Do not file Reconsideration without a fix — repeated rejections delay recovery.
   - Security issue (e.g., "Hacked: Content injection") → treat as an incident. Rotate any exposed credentials, audit Vercel deploys for unauthorized commits, check `next.config.js` and `middleware.ts` for injected code. After remediation, click **Request a review**. Cross-reference with `.claude/rules/cron.md` — verify `CRON_SECRET` wasn't leaked.

### 2.12 Settings

- (a) **What it is.** Property-level configuration and meta-reports.
- (b) **Where to click.** **Settings** (gear icon, bottom of left rail).
- (c) **What to look for.**
   - **Ownership verification** — should show "Verified" with method = "DNS TXT record".
   - **Users and permissions** — only people who should access the property. Remove anyone who no longer needs it.
   - **Crawl stats** (under "Crawling") — graph of Googlebot crawl activity. For a small site, expect 50–500 crawl requests per day. Look for sudden drops (could indicate server errors GSC saw that you didn't) or sudden spikes (could indicate a rogue parameter generating infinite URLs).
   - **robots.txt** (covered in section 1.5) — should be re-verified here.
   - **Change of address** — only relevant if you migrate domains.
- (d) **How to act.**
   - Crawl Stats showing >5% non-200 responses → click the "By response" breakdown. 404s on legitimate URLs mean broken internal links; 500s mean server bugs. Tie to the relevant code area as a TODO.
   - Crawl Stats showing a sudden drop in crawl rate → confirm the site is still reachable, then check robots.txt didn't accidentally disallow more than intended.
   - Stale users in Users and permissions → revoke access.

## Phase 3 — Recurring Cadence

### 3.1 Weekly (5–10 minutes)

Glance, don't analyze. Open GSC, click through this list, and only dig deeper if a threshold below is crossed.

- **Overview screen.** Any red badges on any card? If yes, click through.
- **Performance → Search results, Last 7 days, Dates tab.** Did clicks or impressions drop more than 25% week-over-week? If yes, switch to Queries tab and look for which queries lost the most.
- **Indexing → Pages.** Did "Indexed" count drop by more than 2? If yes, click "Why pages aren't indexed" and check the top reason.
- **Indexing → Sitemaps.** Status = Success? Last read within 7 days? If no, refresh.
- **Enhancements (any schema).** Any new "Invalid" or "Warnings" since last week? If yes, click through.
- **Manual actions + Security issues.** "No issues detected"? (This takes 5 seconds and is the most important check.)

**Ignore at this cadence:**
- Position changes < 3 spots (noise)
- Impressions changes < 25% (noise)
- "Discovered – currently not indexed" counts (this fluctuates; only check monthly)
- Individual query CTRs (only meaningful in 28-day windows)

### 3.2 Monthly (30–45 minutes)

Read deeply, pull exports, compare against prior month.

- **Performance → Search results, Last 28 days.**
  - Compare to **Previous period** (toggle at top): clicks delta, impressions delta, CTR delta, position delta.
  - Compare to **Same period last year** once you have 12+ months of data.
  - Export the **Queries** table (CSV button, top right). Look at the top 50 queries by impressions: which non-brand queries are gaining? Which lost? Are any new queries appearing that suggest new page opportunities?
  - Export the **Pages** table. Identify pages with declining position over 3 consecutive months — these need content refresh or backlink work.
- **Indexing → Pages.**
  - Indexed count trend (should be stable or growing). Export "Not indexed" by reason — particularly track "Crawled – currently not indexed" and "Discovered – currently not indexed" as month-over-month counts.
- **Experience → Core Web Vitals.**
  - URL counts in Good/Needs improvement/Poor. Track Mobile LCP 75th percentile specifically.
- **Enhancements.** All schema reports: Valid count stable, Invalid count = 0.
- **Links → External + Internal.**
  - New referring domains acquired this month. New top-linked pages. Anchor-text drift.
- **Settings → Crawl stats.**
  - Total crawl requests this month. Average response time (should be < 1000ms for a Vercel-hosted site).

**Comparisons to run every month:**
1. Month-over-month: this month's totals vs. last month's, for Clicks / Impressions / Avg position / Indexed pages.
2. 3-month rolling average vs. the prior 3-month rolling average (smooths out noise).
3. Year-over-year (only once Day 365 has passed): this month vs. same month last year — controls for seasonality.

### 3.3 Alert triggers (pull you in regardless of cadence)

Any of these = open GSC the same day, regardless of when you last looked:

- **Email alert from GSC** of any kind (these are pre-triaged by Google; treat them as real).
- **Manual action** issued (also generates an email, but check the report directly).
- **Security issue** detected (also email).
- **Indexed-pages count drops by more than 10%** in a single week (e.g., 27 → 24 or fewer).
- **Sitemap status flips from Success to anything else.**
- **Core Web Vitals "Poor" URL count rises above zero** on Mobile (PH mobile-first market — this is critical).
- **Clicks drop more than 40% week-over-week** with no obvious cause (holiday, content removal, etc.).
- **Average position drops more than 5 spots** in 7 days on any priority commercial query (e.g., "general contractor iloilo").
- **New "Invalid" rows in any Enhancements schema report** (means recent code change broke structured data).
- **Crawl Stats shows >10% 5xx responses** for 3+ consecutive days (means production is intermittently failing for Googlebot).

## Secondary

### S.1 GA4 cross-reference

GSC tells you what queries brought users in. GA4 tells you what they did after. Pair these:

1. **GSC Performance → Search results (Queries tab) × GA4 Reports → Acquisition → Traffic acquisition (filter: Session source = google, Medium = organic).** Pull both for the same 28-day window. The divergence between GSC "Clicks" and GA4 "Sessions where Session source = google / organic" should be small (10–20%). If GA4 sessions are <50% of GSC clicks, GA4 tracking is misfiring on some pages — audit the GA4 install in `src/components/` for SSR/CSR consistency.
2. **GSC Performance → Search results (Pages tab) × GA4 Reports → Engagement → Pages and screens.** Identifies pages with traffic but no engagement: high impressions in GSC, high pageviews in GA4, but low average engagement time → intent mismatch on the page (revise content to match the inbound query). This is the most actionable cross-reference for arsd.co — most likely candidate is `/our-services` ranking for a service the company doesn't actually provide.
3. **GSC Performance (Queries) × GA4 Explore → Free-form with Landing page + Session source dimensions, Conversions metric.** Reveals which queries actually drive contact-form submissions (the only "conversion" on arsd.co). Queries with high impressions but zero conversions = vanity traffic. Queries with low impressions but high conversion rate = priority content investment.
4. **GSC Performance (Devices) × GA4 Reports → Tech → Tech details (Device category).** Confirms PH mobile-first reality and flags any device-specific UX bug (e.g., mobile clicks but mobile bounce rate is 90% → page is broken on mobile).

GA4 conversion setup pre-condition: the `/contact-us` form submission must be tracked as a conversion event. If it isn't, set that up first or none of the conversion-based analyses above will produce signal.

### S.2 Bing Webmaster Tools

Bing matters disproportionately for AI engines: **Copilot, ChatGPT browsing, and DuckDuckGo all draw from Bing's index.** A site invisible in Bing is invisible to a meaningful slice of AI-engine answers. This compounds the **Task 16** GEO work — your GEO content can only be cited by AI engines that found it.

1. **Verification.** Visit `https://www.bing.com/webmasters`. Easiest path: **Import from Google Search Console** (use the same Google account that owns the GSC property; Bing imports site verification + sitemap automatically). Skip the manual DNS dance if this option works — it usually does.
2. **Sitemap submission.** If the import didn't carry the sitemap, manually submit `https://www.arsd.co/sitemap.xml` under **Sitemaps**.
3. **Bing's equivalent of GSC Performance: "Search Performance".** Same axes (queries, pages, impressions, clicks, CTR, position) but Bing's traffic share in PH is small — expect impression volumes ~5–10% of GSC numbers. Useful for catching queries that Bing ranks well and Google doesn't (and vice versa).
4. **Bing's equivalent of GSC Indexing → Pages: "Site Explorer" + "URL Inspection".** Confirm the same 25–27 URLs are indexed by Bing. If Bing's indexed count lags significantly, use **Submit URLs** (Bing allows up to ~10,000 URLs per day, vastly more generous than Google's 10).
5. **IndexNow (Bing-driven protocol).** Bing's "Submit URLs" feature can be automated via the IndexNow protocol — relevant only if you start publishing frequent new content (blog posts, project case studies). Until that happens, manual submission via the UI is enough. Note as a backlog item: if content publishing accelerates, add an IndexNow ping to the Next.js build/deploy pipeline.
